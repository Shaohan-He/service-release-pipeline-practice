# 发布流程

本项目模拟一个业务服务从代码提交到发布上线的完整流程，重点是流程闭环，而不是复杂业务逻辑。

## 主流程

```text
代码提交
  -> GitHub Actions CI
  -> pytest
  -> Docker build 检查
  -> Kustomize render 检查
  -> 构建正式镜像
  -> 推送 version tag 和 commit sha tag
  -> Trivy 镜像扫描
  -> staging 自动部署
  -> staging smoke test
  -> production 手动审批
  -> production canary 部署
  -> canary smoke test
  -> production stable 部署
  -> stable smoke test
  -> 生成 release record
```

## CI 阶段

`ci.yml` 不依赖真实 Kubernetes 集群，也不依赖 secrets。它负责证明代码、镜像构建上下文和 Kustomize 配置至少可以被静态验证。

## 镜像阶段

`build-image.yml` 构建镜像并推送两个 tag：

- 版本号 tag，例如 `v1.1.0`。
- commit sha tag，例如 `abcdef1`。

镜像推送后执行 Trivy 扫描，`HIGH` 和 `CRITICAL` 漏洞会阻塞发布。

## Staging 阶段

staging 代表预发环境。它可以在 main 分支镜像构建成功后自动部署。部署完成后必须等待 rollout 成功，再执行 smoke test。

## Production 阶段

production 是模拟生产环境 namespace。发布通过 `promote-production.yml` 手动触发，并使用 GitHub Environment reviewer 审批。

审批通过后流程先部署 canary，验证通过后再部署 stable。stable 部署成功后生成 release record。

## 异常处理

如果 canary smoke test 失败，不更新 stable。如果 stable 发布后出现问题，使用 `scripts/rollback.sh` 回滚，并在回滚后再次执行 smoke test。
