# 故障演练记录

以下内容是演练样例，用于说明如何记录发布成功和异常回滚过程，不代表真实生产事故。

## 场景一：发布 v1.1.0 成功

### 背景

计划将 `release-demo-service` 从 `v1.0.0` 提升到 `v1.1.0`。该版本只包含接口说明和版本信息调整。

### 操作步骤

1. 合并代码到 main。
2. CI 执行 pytest、docker build 和 Kustomize render。
3. 构建镜像 `v1.1.0` 和 commit sha tag。
4. Trivy 扫描通过。
5. staging 自动部署。
6. staging smoke test 通过。
7. 触发 production promote workflow。
8. production 审批通过。
9. canary 部署并通过 smoke test。
10. stable production 部署并通过 smoke test。
11. 生成 release record。

### 验证点

- `/healthz` 返回 `{"status":"ok"}`。
- `/readyz` 返回 `{"ready":true}`。
- `/version` 显示 `version=v1.1.0`。
- `/api/order` 返回模拟订单。

### 结论

发布成功，release record 已归档。

## 场景二：发布 v1.2.0 后异常并回滚到 v1.1.0

### 背景

计划将 `release-demo-service` 从 `v1.1.0` 提升到 `v1.2.0`。发布后演练异常接口：

```bash
curl -i "http://localhost:8080/api/order?fail=true"
```

该接口返回 500，用于模拟异常版本触发回滚流程。

### 发现

发布后常规 smoke test 通过，但手动演练异常请求时发现订单接口在特定参数下返回 500。确认该行为不符合当前发布预期后，决定回滚到上一稳定版本 `v1.1.0`。

### 回滚操作

```bash
bash scripts/rollback.sh production v1.1.0
bash scripts/smoke-test.sh production
```

### 验证点

- Deployment rollout 成功。
- `/version` 显示 `version=v1.1.0` 或 `image_tag=v1.1.0`。
- `/healthz`、`/readyz`、`/api/order` 正常。
- release record 中记录了上一版本和回滚命令。

### 结论

回滚完成，服务恢复到上一稳定版本。后续需要在修复异常逻辑后重新走 staging 和 canary 验证。
